A draft EDCS document that describes the build strategyfor TPL for NCS.
These documents will describe in detail the tools, components, interactions, and user interface for how each development scrum will utilize the build environment.  In addition the linkage to TPL and management (sustaining effort) will be described.
 
A NCS wide meeting will be held to socialize these approaches and gather any on-going efforts or next steps.

Notes

The expectation is that there will be the following:
 
- Wiki with explanation of the design and choices (this may that the form of a FFS if appropriate).
   - Overview of the build sequence from the perspective of the developer
   - Details on infrastructure requirements
   - Scale / geographic disparite location considerations
- At the end there should be a demo to the development community (that is recorded) to explain the steps and findings.

7: Prototype full build/test of a component

Difference Status: deleted

Formatted ID: US133202

Object ID: 13418532021

Description

Using dbl4j, execute a build, test sequence which updates gerrit with a 0 or -1 if the tests pass. Use the gradle script to spawn the tests

Tasks

add jenkins pre-merge build script

add jenkins post-merge test script

gradle return build result to jenkins

gradle returns test result status to jenkins

Acceptance Criteria

Scenario 1: invoke the gradle build and gradle test when dlbl4j code is pushed:

given dbl4j component with gradle task to execute unit tests,
when the code is pushed to gerrit, then jenkins will execute a component build (gradle jar)
and then if that passes,
then jenkins will invoke the gradle test. 

Scenario 2: On success, jenkins updates gerrit with a +0:
Given that the gradle test passes, then jenkins will receive a success notification
and jenkins will update gerrit with 0.

Scenario 3: On failure, jenkins updates gerrit with -1:
Given that the gradle test fails, jenkins will receive a failure notification
and jenkins will update gerrit with -1
